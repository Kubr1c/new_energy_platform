## 🔄 第三部分：数据流详解

### 用一个真实案例看数据怎么流动

**场景**：用户想查看"今天的电池健康度"

#### 第1步：用户点击页面上的"电池状态"按钮
```
用户界面（前端）
    ↓
HTML按钮被点击
```

#### 第2步：前端发送请求给后端
```
前端 (Vue.js)
    └─ 发送 HTTP 请求：
       GET /api/battery/health
       └─ 这等于说："嘿后端，给我电池的健康数据！"
```

#### 第3步：后端接收请求并处理
```
后端 (Flask)
    ├─ 接收 /api/battery/health 请求
    ├─ 执行 Python 函数：
    │  ├─ 从数据库查询最新电池数据
    │  ├─ 计算健康度（SOH）
    │  └─ 格式化成 JSON（网络格式）
    └─ 返回给前端
```

#### 第4步：后端如何从数据库获取数据
```python
# 简化的代码示例
def get_battery_health():
    # 查询数据库：从"电池状态"表取最新一条记录
    latest_battery = BatteryData.query.order_by(
        BatteryData.timestamp.desc()
    ).first()
    
    # 计算健康度（简化公式）
    soh = (latest_battery.capacity / 100) * 100
    
    # 返回给前端
    return {
        "soh": soh,
        "temperature": latest_battery.temperature,
        "cycles": latest_battery.cycles
    }
```

#### 第5步：前端接收数据并显示
```
后端返回的 JSON 数据
    ↓
前端 (Vue.js)
    ├─ 接收数据
    ├─ 把数据显示在页面上
    │  └─ "SOH: 85%"
    │  └─ "温度: 28°C"
    └─ 用 ECharts 画图表
```

#### 完整的往返流程
```
用户点击按钮
    ↓

    ↓
后端收到请求
    ↓
后端查询数据库
    ↓
后端计算结果
    ↓

    ↓
```

## 🤖 第五部分：AI预测模型详解

### 什么是AI预测模型？

**类比**：就像天气预报员根据过去的天气数据预测明天的天气一样，我们的模型根据过去的电力数据预测未来的风电、光伏和负荷。

### 预测的三个对象

```
系统预测三样东西：

1. 风电功率预测
   └─ "下午2点会有多少风力发电"
   
2. 光伏功率预测  
   └─ "下午2点会有多少太阳能发电"
   
3. 负荷预测
   └─ "下午2点会有多少用电需求"
```

### 系统中有哪些预测模型？

系统提供 **5 种不同的预测模型**，像不同的"预测方法"：

| 模型名 | 复杂度 | 适用场景 | 特点 |
|--------|--------|--------|------|
| **LSTM** | 中等 | 时间序列预测 | 能记住长期的趋势 |
| **GRU** | 中等 | 快速预测 | 比LSTM快一点 |
| **CNN-LSTM** | 高 | 复杂模式 | 能识别局部特征 |
| **Attention-LSTM** | 高 | 精确预测 | 关注重要的数据 |
| **Transformer** | 最高 | 最精确预测 | 最复杂，精度最高 |

### 预测模型怎么工作？（简化流程）

```
输入：过去24小时的历史数据
┌────────────────────────────┐
│ 时间  | 风电 | 光伏 | 负荷   │
├────────────────────────────┤
│ 0点   | 5.2 | 0.0 | 32.1   │
│ 1点   | 5.8 | 0.0 | 28.5   │
│ 2点   | 6.1 | 0.0 | 26.3   │
│ ...   | ... | ... | ...    │
│ 23点  | 4.9 | 3.5 | 35.2   │
└────────────────────────────┘
        ↓
    预处理（清洗数据）
        ↓
    归一化（把数据缩放到 0-1 之间）
        ↓
    输入模型（LSTM/Transformer等）
        ↓
    模型进行复杂的数学计算
    └─ 神经网络的许多层
       └─ 提取特征、学习模式
        ↓
    输出：下一小时的预测
┌──────────────────┐
│ 风电: 5.5 MW    │
│ 光伏: 5.2 MW    │
│ 负荷: 34.6 MW   │
└──────────────────┘
        ↓
    反归一化（转换回原始单位）
        ↓
    显示给用户
```

### LSTM 模型什么是 LSTM？

**类比**：LSTM 就像一个聪明的秘书：
- 📝 能记住重要的过去事件（过去24小时的风电如何波动）
- 🧠 能识别规律（通常下午3点风电较弱）
- 🔮 能根据规律做出预测（预测明天下午3点也会较弱）

```
LSTM 的工作流程（极度简化）：

1. 输入层：接收过去24小时的数据
   └─ [5.2, 5.8, 6.1, ..., 4.9]

2. 细胞态（Cell State）：长期记忆
   └─ 记住"下午通常会有风"这样的长期规律

3. 隐藏态（Hidden State）：短期记忆
   └─ 记住"最近几小时的趋势"

4. 门控机制（就像门一样）：
   ├─ 遗忘门：决定忘掉哪些旧信息
   ├─ 输入门：决定接收哪些新信息
   └─ 输出门：决定输出什么

5. 输出层：预测下一小时
   └─ 5.5 MW
```

### 如何使用预测模型

```python
# 代码示例：调用预测模型

from models.predict import Predictor

# 1. 创建预测器（选择模型类型）
predictor = Predictor(model_type='attention_lstm')

# 2. 获取最近24小时的历史数据
recent_data = NewEnergyData.query.order_by(
    NewEnergyData.timestamp.desc()
).limit(24).all()

# 3. 运行预测
prediction = predictor.predict(recent_data)

# 4. 查看结果
print(f"预测结果:")
print(f"  风电: {prediction['wind_power']:.2f} MW")
print(f"  光伏: {prediction['pv_power']:.2f} MW")
print(f"  负荷: {prediction['load']:.2f} MW")
```

### 预测准确度怎么衡量？

系统使用 **MAPE**（平均绝对百分比误差）来衡量：

```
MAPE = (|预测值 - 真实值| / |真实值|) × 100%

例如：
真实值：100 MW
预测值：105 MW
MAPE = (|105 - 100| / 100) × 100% = 5%

MAPE 越小越好：
- < 5%  : 非常准确 ✓
- 5-10%: 准确 ✓
- 10-20%: 可以接受
- > 20%: 需要改进
```

## ⚙️ 第六部分：能量调度优化详解

### 什么是能量调度？

**场景**：你是一个电力交易员，需要决定：
- 🔌 什么时候充电（便宜的时候）
- 🔋 什么时候放电（贵的时候）
- 💰 怎样才能赚最多的钱

这就是"能量调度"要做的事。

### 调度过程（5步）

```
第1步：收集信息
├─ 过去24小时的风电、光伏、负荷数据
├─ 未来24小时的价格预测
├─ 当前电池的电量(SOC)
└─ 电池的参数（容量、效率等）

第2步：预测未来
├─ 预测下24小时会有多少风电
├─ 预测下24小时会有多少光伏
├─ 预测下24小时会有多少负荷
└─ 预测下24小时的电价走势

第3步：运行优化算法
├─ 粒子群优化(PSO)
├─ 遗传算法(GA)
└─ 自适应加权PSO(AWPSO)
    └─ 找到"最优的充放电方案"

第4步：输出充放电计划
├─ 0点-1点：充电5MW
├─ 2点-3点：停止（等待便宜价格）
├─ ...
├─ 18点-19点：放电20MW（价格贵）
└─ ...

第5步：实时执行
├─ 按照计划充放电
├─ 实时监控与调整
└─ 记录实际收益
```

### 优化算法简介

#### PSO（粒子群优化）

**类比**：就像一群鸟一起找食物：
- 🐦 每只鸟都在空间中飞行（代表一个解）
- 🐦 每只鸟记得自己找到的最好位置（个体最优）
- 🐦 每只鸟也知道整群找到的最好位置（群体最优）
- 🐦 每只鸟会向这两个位置靠近，最终找到食物（全局最优）

```
初始状态：
[鸟1] [鸟2] [鸟3] ... [鸟n]
都在随机位置飞行，寻找"最优充放电方案"

迭代过程（重复50-100次）：
├─ 每只鸟评估自己当前位置的好坏
│  └─ 好 = 能赚更多钱，能源浪费少
├─ 每只鸟记住自己的最好位置
├─ 每只鸟看看其他鸟找到什么
├─ 每只鸟向最好位置移动一点点
└─ 重复...

最终收敛：
所有鸟都聚集到最好的位置
这个位置就是"最优充放电方案"
```

#### GA（遗传算法）

**类比**：就像生物进化一样：
- 🧬 一个"充放电方案"就是一条"基因"
- 🧬 好的方案会"繁殖"（产生相似的方案）
- 🧬 坏的方案会"淘汰"（被删除）
- 🧬 经过几代演化，最终产生最优方案

```
第一代：随机生成100个充放电方案
├─ 方案1：赚钱3000元
├─ 方案2：赚钱4500元
├─ 方案3：赚钱3200元
└─ ...

评估+选择：找出最好的50个方案

繁殖：从好的方案里产生新方案
├─ 方案1的子代：赚钱4800元
├─ 方案2的子代：赚钱4600元
└─ ...

第二代：100个新方案
└─ 平均质量变好了

重复多次...

第N代：产生最优方案
└─ 赚钱5200元（比第一代最好的还好）
```

### 调度的约束条件

优化过程需要遵守一些"规则"（约束）：

```
约束1：电池不能过充也不能过放
└─ SOC（电池电量）必须在 10% - 90% 之间
   不能充到 100%（伤电池）
   不能放到 0%（没有应急电力）

约束2：充放功率有限制
├─ 最大充电功率：20 MW
├─ 最大放电功率：20 MW
└─ 一小时内不能超这个值

约束3：与电网通信限制
└─ 最多能往电网送 30 MW 的电
   （电网的接收能力有限）

约束4：物理效率限制
├─ 充电效率：95%（充进去100度电，实际存95度）
├─ 放电效率：95%（放出来100度电，实际用95度）
└─ 往返效率：95% × 95% = 90.25%

这些约束确保：
✓ 电池不损伤
✓ 安全稳定
✓ 电网不超载
```

### 当你想添加一个新功能时

假设你想添加"电池温度告警"功能：

#### 步骤1：后端数据库
```python
# 修改 backend/models/database.py
class BatteryAlert(db.Model):
    __tablename__ = 'battery_alert'
    id = db.Column(db.Integer, primary_key=True)
    temperature = db.Column(db.Float)
    threshold = db.Column(db.Float)
    alert_type = db.Column(db.String(20))  # 'high_temp'/'low_temp'
    created_at = db.Column(db.DateTime, default=datetime.utcnow)
```

#### 步骤2：后端API
```python
# 在 backend/routes/ 中新建 alert.py
from flask import Blueprint, jsonify
from models.database import BatteryAlert, db

alert_bp = Blueprint('alert', __name__, url_prefix='/alert')

@alert_bp.route('/get_alerts', methods=['GET'])
def get_alerts():
    alerts = BatteryAlert.query.all()
    return jsonify({
        'code': 200,
        'data': [
            {
                'id': a.id,
                'temperature': a.temperature,
                'alert_type': a.alert_type
            } for a in alerts
        ]
    })

# 然后在 backend/app.py 注册这个蓝图
app.register_blueprint(alert_bp)
```

#### 步骤3：前端页面
```vue
<!-- 在 frontend/src/views/ 中新建 Alert.vue -->
<template>
  <div class="alert-container">
    <h2>电池告警</h2>
    <table>
      <tr v-for="alert in alerts" :key="alert.id">
        <td>{{ alert.temperature }}°C</td>
        <td>{{ alert.alert_type }}</td>
      </tr>
    </table>
  </div>
</template>

<script>
export default {
  data() {
    return { alerts: [] }
  },
  mounted() {
    // 获取告警数据
    fetch('/alert/get_alerts')
      .then(res => res.json())
      .then(data => { this.alerts = data.data })
  }
}
</script>
```

#### 步骤4：添加路由
```javascript
// 在 frontend/src/router/index.js 中添加
{
  path: '/alert',
  name: 'Alert',
  component: () => import('../views/Alert.vue')
}
```
